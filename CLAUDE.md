# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Felix — a **managed agents harness** on Cloudflare Workers, in the shape Anthropic describes in [Managed Agents](https://www.anthropic.com/engineering/managed-agents). The harness virtualizes three abstractions so each can be swapped without disturbing the others:

- **Session** — an append-only event log per thread. The harness asks a `SessionStrategy` (`full_replay` / `windowed:N` / `summarizing:N` / `semantic:N`) to render the working-set message array for each model call instead of mutating an in-memory history. Anchor messages (`metadata.pinned: true`) are always included across strategies.
- **Pattern / model registries** — `react`, `deep`, `router`, `parallel`, `groupchat`, `reflect`, `plan_execute` and the Anthropic / OpenAI / Workers AI providers self-register at module load; `getPattern(name)` / `getModelProvider(name)` resolve them at build time. Adding a new loop or provider is one `register*(...)` call.
- **ToolExecutor seam** — every `Tool` carries a `transport`-labelled executor (`local` / `mcp` / `a2a` / `container` / `queue` / `sandbox` / `browser`). The model loop dispatches by name; the harness routes to whichever transport the tool was built with. `queue` is the async transport: the executor enqueues a job carrying `tool_call_id`, returns a chatty stub, and a separate consumer writes the eventual `tool_result` back to the session via `POST /internal/sessions/:thread_id/events` (`packages/harness/src/api/internal.ts`) — `wake()` + `tasks/resubscribe` make the resume seamless. That write-back is gated by the fleet-global `CONSUMER_SHARED_SECRET` (constant-time) **plus dispatch pairing**: each `tool_result` must match an outstanding `queue_dispatch` audit row for that tenant + thread and not already be resolved (`queue_complete`/`queue_expired`), else 409 with nothing written — closing the cross-tenant forged-write-back vector; the emitted `queue_complete` carries the paired dispatch's manifest id. `sandbox` (Cloudflare Sandbox SDK) and `browser` (Browser Rendering via `@cloudflare/puppeteer`) front separate Worker adapters as Service bindings; reference adapters live in `examples/sandbox-worker/` and `examples/browser-worker/`.

A `apiVersion: orchestrator/v1` manifest declares the agent — pattern, model (with optional `fallbacks` + `confidence_escalation`), system prompt, tools, skills, MCP servers, A2A peers, containers, queues, sandboxes, browser tools, memory backend, session strategy, execution mode (transient/durable), tools_retrieval, artifacts, reflect config, procedural memory, auth, policies, limits, guardrails (incl. `judges[]`), approvals — and `buildAgent(...)` compiles it to a runnable `Agent`. The runtime is TypeScript on Hono, with Durable Objects, Neon Postgres via Hyperdrive (pgvector included), R2, KV, Queues, Workflows, Analytics Engine, and AI Gateway. Public surfaces: `/v1/*` (OpenAI-compatible), `/chat`, `/a2a`, `/mcp`, `/jobs`, `/plans`, `/audit` (+ `/audit/metrics`), `/approvals`, `/manifests` (+ `/:name/canary`, `/:name/rollback`), `/eval`, `/.well-known/agent-card.json`, `/.well-known/jwks.json` (self-issued JWKS when `JWKS_PUBLIC` set), plus the commerce mounts: `/commerce` (Stripe webhook, consents, attribution), `/acp` (Agentic Commerce Protocol; `ACP_API_KEY` bearer, skips JWT middleware), `/ucp` + root `/.well-known/ucp` (Universal Commerce Protocol; `UCP_API_KEY` bearer, skips JWT middleware), `/brands`, `/shop` + `/widget` (public per-brand storefront + embed), `/structured` + root `robots.txt` / `sitemap.xml` / `.well-known/ai-catalog.json`, `/entities`, `/b2b` (+ `/b2b/billing`), `/geo`. Manifests can be bundled at build time, dropped into R2, or managed per-tenant through the `/manifests` REST surface (append-only versions with active-pointer rollback, optional canary pointers; gated by `manifests:read` / `manifests:write` scopes). Every management route requires a per-surface scope via `requireScope` (`audit:read`, `approvals:read`/`approvals:decide`, `plans:read`, `jobs:read`/`jobs:write`, `eval:read`/`eval:write`, `consent:read`, `geo:write`, `brands:write`, `b2b:write`, `entities:write`); the gate falls open only in dev with no verifiers.

**Felix Commerce** — the agentic-commerce layer, a separate workspace package: **`@felix/commerce`** at `packages/commerce/` (commerce + `entities/` + `geo/` under its `src/`). Built on the harness seams: catalog/cart/checkout tools (cart lives in the session log as a pinned `audit` event, checkout is approval-gated Stripe), ACP merchant endpoint, per-brand D2C storefront manifests + widget, schema.org/GEO discoverability, B2B quote-to-cash on a pluggable entity data-source seam (native/federated/synced; http/mcp connectors), personalization + visual search + dynamic pricing, consent + attribution. The package exports one supported symbol — a **`FelixPlugin`** (`packages/commerce/src/plugin.ts`; contract in `packages/harness/src/plugins/types.ts`) bundling routers, tool factories, cron tasks, the `/acp` self-auth mount, rate-limit keying, and the body-size floor — and the `@felix/api` Worker wires it in through the single `commercePlugin` entry in `apps/api/src/composition.ts:installedPlugins()`. The package consumes harness seams via `@felix/harness/<path>` source exports (no build step; wrangler/vitest/tsc all resolve TS source through package `exports`); commerce env vars merge into `Env` via module augmentation in `packages/commerce/src/env.ts`. The harness itself is completely commerce-blind (enforced by `apps/api/tests/unit/plugin_boundary.test.ts`). Detail in `packages/commerce/docs/index.md`.

## Architecture

### Wiring entry points
The deployable Worker shell is `apps/api` (@felix/api); the harness library is `packages/harness` (@felix/harness).
- `apps/api/src/index.ts` — Worker default export: `fetch`, `scheduled` (cron: federation refresh + `runScheduledJobs` + `sweepOrphanQueueDispatches` + `runAnomalyScan` + `runContinuousEvalTick`, then every installed plugin's `cronTasks` — commerce contributes `abandoned_cart_scan` and `geo_monitor_tick` — each isolated in its own try/catch), `queue` (audit batching). Also re-exports every DO class (`ConversationDO`, `A2ATaskDO`, `ApprovalsDO`, `FederationDO`) and `AgentWorkflow` (`apps/api/src/agent-workflow.ts`; its params type stayed in the harness at `packages/harness/src/workflows/types.ts`) — wrangler.jsonc references these names.
- `packages/harness/src/app.ts` — `createApp({ tools, defaultManifest, plugins })` returns a Hono app. Receives the `ToolProvider` by injection so tests swap in a stub. Mounts the core sub-routers plus `/health` and the agent-card endpoint, then calls each plugin's `routes(app, { tools })` (before `/docs`); plugins also contribute the auth middleware's `selfAuthenticatingMounts`, the rate limiter's `keyResolvers`, and the `bodyLimit` ceiling (`max(1 MB core floor, ...plugin bodyLimitBytes)`). Also registers the unhandled-exception boundary (`app.onError`): `HTTPException`s thrown by routes flow through with their own response; anything else is turned into a 500 `{error:{message:'internal error'}}`, an `orchestrator_unhandled_error` counter, a structured `unhandled_exception` error log (with stack), **and** an `unhandled_error` audit event so tenant operators see it in `/audit` alongside everything else.
- `apps/api/src/composition.ts` — the only place that knows about both the tool catalog and the harness's `ToolProvider` abstraction. Adding a tool = `provider.register('name', () => defineTool({...}))` here; no harness changes needed. Skill-activation tools (`list_skills` / `activate_skill` / `deactivate_skill`) live here because they read `RequestContext`. Also the plugin wiring root: `installedPlugins()` returns the `FelixPlugin[]` (currently just `commercePlugin`) — the ONLY line outside `packages/commerce` that names a plugin; `compose` registers each plugin's tools through it.
- `packages/harness/src/env.ts` — single source of truth for binding types. Mirrors `apps/api/wrangler.jsonc`. `parseModelRoutes(env)` parses the `MODEL_ROUTES` JSON map.

### Request-scoped state (AsyncLocalStorage, not parameter threading)
`packages/harness/src/context.ts` defines `RequestContext` ({env, auth, limitState, threadId, manifestId, execCtx}). `authMiddleware` (`packages/harness/src/auth/middleware.ts`) populates it via `runWithContext` before any route runs and tears it down with `disposeLimitState` in `finally`. Tools and governance wrappers read it through `getContext()` / `requireContext()`. Preserve this pattern when adding cross-cutting concerns — tool authors should never see limit/policy state as an explicit parameter.

`LimitState` carries an `AbortController` that fires on wall-clock breach or request teardown. The signal is plumbed through `ToolInvocationCtx.signal` (peer + MCP fetches honor it) and through `ModelChatOptions.signal` (Anthropic + OpenAI gateway calls honor it). It also carries cumulative `tokens: { input, output }` accumulated from every model response — the react loop checks `checkTokenBudget(limits, manifestId)` before each model call to enforce `max_input_tokens` / `max_output_tokens` caps.

Cron handlers run **outside** `authMiddleware`. `index.ts:scheduled` wraps its body in `runWithContext(buildAnonymousContext(env, ctx), …)` + `disposeLimitState` so audit events actually persist (recordEvent falls back to `console.log` when no context is installed). The queue consumer does not install a RequestContext — it only persists `AuditEvent` rows that were already enqueued by request-scoped code, with a per-row fallback if the batch insert fails so one poison row doesn't starve the queue.

`app.onError` also runs outside any `runWithContext` scope: the auth middleware's `runWithContext` has already unwound by the time Hono routes the exception to the error handler, so `getContext()` returns undefined there. The handler instead uses `recordEventDetached(env, opts, execCtx)` (`packages/harness/src/audit/store.ts`) — same pipeline as `recordEvent`, but takes `env` and `execCtx` explicitly off the Hono context instead of reaching for AsyncLocalStorage. Use this helper for any other detached callers that need to emit an audit event.

### Manifest → Agent pipeline
`packages/harness/src/manifests/builder.ts:buildAgent` is the single compile step:
1. load + Zod-validate (`schema.ts`, `validate.ts`). `validate.ts` queries `isMultiAgentPattern(name)` on the pattern registry, so cross-field rules (sub_agents required, peers forbidden, durable mode requires single-agent + checkpointed memory, sandboxes/browser_tools forbidden on multi-agent) auto-apply.
2. resolve system prompt (soul / inline / base)
3. resolve `SessionStore` (`getSessionStore(env, memory.checkpointer)`) and `SessionStrategy` (`getSessionStrategy(spec.session.strategy)`)
4. compose skills (`packages/harness/src/skills/loader.ts` + per-tenant activation overlay from Postgres)
5. bind external MCP servers (namespaced) — each remote tool gets an `McpExecutor` (`transport: 'mcp'`)
6. build A2A peer tools (`a2a/client.ts:makePeerTool`) — each gets an `A2AExecutor` (`transport: 'a2a'`); `mcp/client.ts` forwards remote MCP `inputSchema` to the LLM via the `rawInputSchema` escape hatch on `Tool`
7. build container tools (`tools/container-executor.ts:makeContainerTool`) — each `spec.containers[]` entry becomes a tool with a `ContainerExecutor` (`transport: 'container'`); the same auth broker MCP / A2A use supplies the `Authorization` header so raw tokens never enter the executor closure
8. build queue tools (`tools/queue-executor.ts:makeQueueTool`) — each `spec.queues[]` entry resolves a Cloudflare Queue producer from `env[binding]` and wraps it in a `QueueExecutor` (`transport: 'queue'`). Missing bindings fail the build.
8b. build sandbox tools (`tools/sandbox-executor.ts:makeSandboxTool`) — each `spec.sandboxes[]` entry wraps a Service binding in a `SandboxExecutor` (`transport: 'sandbox'`). The bound Worker fronts the Cloudflare Sandbox SDK (see `examples/sandbox-worker/`).
8c. build browser tools (`tools/browser-executor.ts:makeBrowserTool`) — each `spec.browser_tools[]` entry wraps a Service binding in a `BrowserExecutor` (`transport: 'browser'`). The bound Worker fronts `@cloudflare/puppeteer` over Browser Rendering (see `examples/browser-worker/`).
9. resolve sub-agents (multi-agent patterns) and tools (everyone else); auto-inject memory tools when `memory.store` resolves to `vectorize`, `recall_procedure` when `spec.procedural_memory.enabled`, `fetch_artifact` when `spec.artifacts.enabled`, and `PLAN_TOOLS` (`plan_create`/`plan_update_step`/`plan_get`) when `spec.pattern === 'deep'`. All of these are injected BEFORE the governance pipeline so they are gated by policies/limits/guardrails/judges/approvals like any other tool; `patterns/deep.ts` no longer re-injects them.
10. governance pipeline: `mergeWithManifest` → `applyPolicies` → `applyLimits` → `applyGuardrails` → `applyJudges` → `applyApprovals`. Each wrapper replaces `tool.executor` via `wrapExecutor(...)` so the inner transport label survives.
11. `getPattern(spec.pattern)(ctx)` — open registry dispatch. The pattern descriptor picks what it needs from `PatternBuildContext` (`tools`, `subAgents`, `session*`, `limits`, `toolsRetrieval`, `artifacts`, …).
12. if `spec.execution.mode === 'durable'`, wrap the result via `wrapDurableAgent(agent, manifest, env)` — the wrap runs `invoke` inside `env.AGENT_WORKFLOW.create({ params })` so a Worker eviction mid-loop replays the step. Falls back to transient when the binding is absent.

Builder takes `BuildDeps` (env, tools, optional auth/soulLoader/extraTools/subAgentBuilder) — don't reach for module-level singletons. Adding a new pattern or provider is a `registerPattern(...)` / `registerModelProvider(...)` call in the relevant module's bottom matter; `composition.ts` can register additional ones at deploy time without core changes.

### Patterns
No LangGraph. `patterns/react.ts` is a hand-written tool loop bounded by `recursion_limit` (model turns) + the limits wrapper (per-call caps: tool calls, peer hops, wall clock, tokens). Tool dispatch is sequential for deterministic audit ordering. Tools that throw a `ToolError` (`packages/harness/src/tools/errors.ts`) are stringified as `[tool error/<code>] <msg>` (code ∈ `invalid_arguments` | `transport_unavailable` | `provider_error` | `timeout` | `user_aborted` | `rate_limited` | `permission_denied` | `internal`) and fed back to the model unless the tool declares `fatal: true` — those terminate the loop with the error message as `final`. Multi-agent patterns each get a dedicated DO for state. Router and parallel call `checkTokenBudget(limits, manifestId)` before each child dispatch; react calls `checkPreflightTokenBudget(...)` (Anthropic's free `/v1/messages/count_tokens`) and falls back to `checkTokenBudget` for routes that don't implement `countTokens`. When `spec.tools_retrieval.enabled`, react filters tools by BGE cosine similarity before each model call. When `spec.artifacts.enabled`, tool results over `threshold_chars` are spilled to R2 (`fetch_artifact` reads them back).

`reflect` (`patterns/reflect.ts`) wraps a react base with a verifier model; below `spec.reflect.threshold` it appends the critique as a synthetic user turn and replays react up to `spec.reflect.max_iterations`. Emits `judge_score` audit events with `payload.source: 'reflect'`. `plan_execute` (`patterns/plan-execute.ts`) splits the loop into a planner model (emits a JSON subtask list), an executor model (runs each subtask in a bounded react sub-loop), and a synthesizer pass (produces the final assistant turn). Emits `plan_step` audit rows with `payload.source: 'plan_execute'`; re-plans up to `spec.plan_execute.max_replans` times when a subtask fails and `replan_on_failure: true`.

`patterns/model.ts` is the AI Gateway model client — Anthropic + OpenAI through `https://gateway.ai.cloudflare.com/v1/${AI_GATEWAY_ACCOUNT_ID}/${AI_GATEWAY_SLUG}/<provider>/...`, Workers AI through the native `env.AI` binding. Every gateway call attaches `cf-aig-authorization: Bearer ${CF_AIG_TOKEN}` when the secret is set (required for slugs with Authenticated Gateway enabled). `streamChat` returns `AsyncGenerator<string, ModelChatResult>` — text deltas yield, the final assistant turn (with any `tool_calls`, `thinking` / `redacted_thinking` blocks, and `usage`) is the generator's return value. The streaming react loop captures it with `await stream.next()` once `done === true`, eliminating the second non-stream `chat()` call earlier versions made. Workers AI SSE reads are buffered across chunk boundaries so split frames don't drop deltas.

`spec.model.fallbacks: string[]` wraps the primary client with `withModelFallbacks` — on `provider_error`/`rate_limited`/`timeout` the chain replays the same call against each fallback id, emits a `model_switch { reason: 'provider_error' }` audit, increments `orchestrator_model_switches`. `spec.model.confidence_escalation: { enabled, escalate_to, low_confidence_markers, min_response_chars }` routes low-confidence assistant turns to a higher-capacity model (`reason: 'low_confidence'`).

Anthropic-specific knobs on `spec.model`: `cache: true` adds `cache_control: { type: 'ephemeral' }` markers to the system prompt, the last tool definition, and the last conversation message (skipped on `thinking` / `redacted_thinking` blocks — Anthropic rejects cache markers there). `thinking_budget` (1024–64000) enables extended thinking, forces `temperature: 1`, and bumps `max_tokens` to at least `budget + 1024`; returned `thinking` / `redacted_thinking` blocks are captured on the assistant message and round-tripped on the next request so tool-result follow-ups don't get rejected. Both are no-ops for OpenAI / Workers AI routes (OpenAI prompt caching is automatic — its `cached_tokens` are subtracted from `prompt_tokens` before populating `TokenUsage.input` so they don't double-count against `max_input_tokens`).

### Session persistence
The `Session` abstraction (`packages/harness/src/session/types.ts`) is the harness's external context object. Each `SessionEvent` carries a monotonic `seq`, a `kind` discriminator (`message` / `tool_result` / `tool_call` / `thinking` / `audit`), the message-shaped payload, and optional `metadata`. `ConversationDO` (`packages/harness/src/memory/conversation-do.ts`) is keyed by thread id and exposes `GET /events?from&to&limit&kinds`, `GET /head`, `POST /events`, `DELETE /events` — there is no all-or-nothing `/history` dump.

`SessionStore.open(threadId)` returns a per-thread `Session`. The react/deep/groupchat patterns call `await strategy.render(session, incoming, { systemPrompt, model })` to produce the working-set message array, then append new events incrementally via `persistFireAndForget(session, events, { manifestId })` (in `packages/harness/src/session/do-session.ts`), which routes through `execCtx.waitUntil` so DO writes don't block LLM steps. Bounded retry (3×, 50ms / 150ms / 450ms) covers transient failures; terminal failures emit a `checkpoint_failure` audit event + `orchestrator_checkpoint_failures` counter rather than being swallowed by `console.warn`.

`SessionStrategy` (`packages/harness/src/session/strategies.ts`) decides what the model sees:
- `full_replay` (default) — every prior message, exactly as in the legacy checkpointer.
- `windowed:N` — keep the last N events; drop the rest.
- `summarizing:N` — keep the last N raw events, model-summarize everything older into a synthetic system message. The summary is cached as a `kind: 'audit'` event with `metadata: { type: 'session_summary', covers_to_seq: N }`, so steady-state rendering skips the model call until new events cross the keep boundary. Degrades to windowed when no model is supplied or the summarizer call throws — never fails the request.
- `semantic:N` (`packages/harness/src/session/semantic-strategy.ts`) — embed the current user message via BGE and pull the top-N most relevant prior events from the session log instead of the most recent N. Anchor messages (`metadata.pinned === true`) are always included regardless of score.

`Session.wake()` (`packages/harness/src/session/types.ts:analyzeWake`) is the resume primitive — Anthropic's Managed Agents `wake(sessionId)`. Read-only analysis of the event log that returns `{ fresh, headSeq, pendingToolCalls, endedOnAssistant }`. Used by the A2A `tasks/resubscribe` method (`packages/harness/src/a2a/server.ts:tasksResubscribe`) to let a reconnecting client pick up: terminal tasks replay events + emit cached output; in-progress tasks replay events + emit a `continue_hint` directing the client to issue `tasks/sendSubscribe` with the same task id so the next loop step picks up from the resume point.

Routing:
- `POST /chat`, `POST /chat/stream` — caller supplies `thread_id` in the body.
- `POST /v1/chat/completions` — opt-in via `x-thread-id` header; absent, each request is stateless (`tenant:openai-<uuid>`).
- A2A `tasks/send` / `sendSubscribe` — the task id becomes the thread id (`tenant:a2a-<taskId>`).
- **router** forwards `threadId` to its chosen child. **parallel** strips it (prevents race writes). **groupchat** owns the transcript itself: hydrates from + appends to the parent threadId, deliberately does **not** forward it to children (they'd race-write the same DO).

### Persistence layout
- Postgres (`HYPERDRIVE` → Neon; schema `apps/api/migrations/0001_baseline.sql`, node-pg-migrate) — harness core: audit_events, plans, jobs, approvals (TTL columns + partial unique signature index over live decisions `pending`/`approved`/`denied` so consumed/expired grants are archived and re-requestable), skill_activation, oauth_token_cache, manifests + manifest_active (canary_version/canary_weight; resolver picks stable vs canary by SHA-256 hash of `(tenant_id, thread_id, manifest_name, stable_version, canary_version)`), eval_datasets + eval_dataset_items + eval_runs (`manifest_version` powers the opt-in eval activation gate on `POST /manifests/:name/activate` + `/canary`). Commerce: products (with generated `search_tsv` weighted FTS column + pg_trgm title index backing `catalog_search`), orders/order_items, acp_checkout_sessions, ucp_checkout_sessions, brands + brand_domains, data_sources, B2B accounts/buyers/quotes/quote_items/invoices/contract_prices/billing_settings, geo_queries/geo_observations, consents/order_attribution, customers/customer_sessions/behavior_events/abandoned_carts, pricing_rules/competitor_prices. Vectors: memory_vectors (768-dim pgvector, HNSW cosine; kinds `fact`/`preference`/`episode`/`procedural`/`product`/`product_image`, scoped by real tenant_id/kind/manifest_id columns). **Tenant-first composite primary keys** (`(tenant_id, id)` in the common case; natural composites like `(tenant_id, name, version)` for manifests; `brand_domains` is host-keyed by design) with `(tenant_id, ts DESC)` indexes — always include the tenant in WHERE clauses; single-column `idx_audit_ts` / partial `idx_plans_expires` exist only for the tenant-agnostic retention sweep. Dialect conventions: timestamps BIGINT epoch-ms (getDb parses int8 → Number), booleans real BOOLEAN, `*_json` columns JSONB queried with `->>`. **Access is only via `getDb(env)`** (`packages/harness/src/db/client.ts`): postgres.js tagged templates, per-request client cached on RequestContext and closed at teardown outside staging/prod (deployed Workers tear sockets down themselves), `sql.begin` for multi-statement writes (manifest activate/purge, order+items, quote+items), `result.count` for row-count control flow (approvals CAS). jsonb params are passed as RAW objects — never pre-`JSON.stringify` (postgres.js Describes the statement and serializes once; double-encoding lands a jsonb string scalar). The Hyperdrive config is created with `--caching-disabled` (read-after-write everywhere) against Neon's DIRECT endpoint — Hyperdrive replaces PgBouncer. An OPTIONAL second caching-enabled config (`HYPERDRIVE_CACHED`, via `getCachedDb`/`withCachedDb`) serves the public read-only surfaces (structured feeds, storefront config); absent, they fall back to the default client. Locally, `wrangler dev` routes the binding through `localConnectionString` (Docker pg). `oauth_token_cache.access_token` is AES-256-GCM encrypted at rest via `OAUTH_CACHE_KEY` (`packages/harness/src/security/at-rest.ts`); dev falls back to plaintext with a warning, staging/production require the secret.
- KV (`CACHE`) — JWKS cache, outbound OAuth token cache, manifest cache.
- R2 (`BUNDLES`) — federation `PolicyBundle`, per-tenant manifest overrides at `manifests/<tenant_id>/<name>.json`, global manifest overrides at `manifests/<name>.json`, and (when `spec.artifacts.enabled`) artifact spills at `artifacts/<tenant_id>/<thread_id>/<tool_call_id>.txt`. Tenant Postgres wins over tenant R2 wins over global R2 wins over bundled. Artifact spills are GC'd by the `retention_sweep` cron: objects whose R2 `uploaded` timestamp is older than `ARTIFACT_RETENTION_DAYS` (parsed by `parseArtifactRetentionDays`; default 30, clamped `[1, 3650]`) are deleted, bounded per tick by both the 5000-object delete cap and a 20-page `list` scan cap (artifact keys carry no time ordering).
- pgvector (`memory_vectors` table, part of the Postgres schema above) — semantic memory, procedural memory, and the commerce product/visual embeddings, all through `packages/harness/src/db/vectors.ts` (`upsertVector`/`queryVectors`/`getVectorValues`/`deleteVector`; score = 1 − cosine distance). Every query names an explicit kind list so pools never bleed. The builder auto-injects `memory_remember` / `memory_recall` when a manifest declares `memory.store: vectorize` (legacy enum name; `agentcore` is the older alias — both resolve to the pg store), and `recall_procedure` when `spec.procedural_memory.enabled`. The `semantic:N` session strategy and JIT tools-retrieval embed inline via `env.AI` (JS cosine) — they do not touch the vector table. Opt-in GC via `MEMORY_RETENTION_DAYS` (unset = keep forever).
- Queue (`AUDIT_QUEUE`) — audit events fan out via the producer in `audit/store.ts`; the batched consumer in `index.ts:queue` flushes ≤50 events per pull in one multi-row INSERT (running under an anonymous RequestContext so getDb caches one client per batch). The same `queue()` handler also drains the `felix-audit-dlq-*` dead-letter queues (branch keyed on the `-dlq` queue name → `drainAuditDlq` in `jobs/audit-dlq.ts`): events the main consumer exhausted its retries on get a best-effort direct re-write plus an `orchestrator_audit_dlq_received` counter, then are acked unconditionally (a DLQ has no further dead-letter). The DLQ consumers are declared alongside the main audit consumer in `wrangler.example.jsonc` (staging/prod only; dev has no DLQ).
- Analytics Engine (`METRICS`) — `recordCounter`/`recordHistogram` write to `env.METRICS.writeDataPoint(...)` when the binding is present; stdout fallback continues for dev parity.
- Workflows (`AGENT_WORKFLOW`) — when a manifest declares `execution.mode: durable`, the builder wraps the agent in a `DurableAgent` that runs `invoke` inside `env.AGENT_WORKFLOW.create({ params })`. The workflow class is `AgentWorkflow` in `apps/api/src/agent-workflow.ts` (params type in `packages/harness/src/workflows/types.ts`).

### Durable Objects
- `ConversationDO` — one per thread id. Carries an idle-TTL alarm: every append sets/renews `storage.setAlarm(now + CONVERSATION_IDLE_TTL_DAYS)` (parsed by `parseConversationIdleTtlDays`; default 90, floored + clamped to `[1, 3650]`); the `alarm()` handler wipes the thread's storage once it's been idle ≥ TTL (emitting `orchestrator_conversation_idle_expired`) and otherwise reschedules to the exact expiry — day-2 GC so a thread doesn't live forever behind `MAX_STORED_EVENTS`.
- `A2ATaskDO` — one per `tenant#task` id, owns A2A task lifecycle state.
- `ApprovalsDO` — serializes concurrent `decide` writes on the same approval id; `/approvals/:id/decide` routes through it.
- `FederationDO` — cron-refreshed singleton holding the active `PolicyBundle` from R2.

### Auth
Inbound JWT verifiers (`auth/jwt.ts`) are configured entirely through the single `JWT_VERIFIERS` env var — there are no provider-specific vars. It's a comma-separated list; each verifier is whitespace-separated `<scheme> <issuer> [audience]` where `scheme` is `access` (Cloudflare Access JWKS at `https://<issuer>/cdn-cgi/access/certs`) or `cognito` (standard OIDC JWKS at `<issuer>/.well-known/jwks.json`). `parseVerifiers(env)` skips malformed/unknown-scheme entries; an empty value means no verifiers (fail-closed in prod, anonymous in dev). Anonymous context is always populated — the route/manifest decides whether `allow_anonymous` is OK. Outbound OAuth client-credentials tokens are cached in Postgres (encrypted at rest, see persistence layout).

### Cron + jobs
`triggers.crons: ["*/10 * * * *"]` → `scheduled()` runs federation refresh, then `runScheduledJobs` (`jobs/cron.ts`), then `runRetentionSweep` (`jobs/retention.ts`) — the day-2 GC pass that prunes `audit_events` older than `AUDIT_RETENTION_DAYS` (parsed by `parseAuditRetentionDays`; default 90, floored + clamped to `[7, 3650]`), expired `plans` (`expires_at < now`), and R2 artifact spills older than `ARTIFACT_RETENTION_DAYS` (`parseArtifactRetentionDays`; default 30, clamped `[1, 3650]`); plus OPT-IN `memory_vectors` rows older than `MEMORY_RETENTION_DAYS` (`parseMemoryRetentionDays`; unset = keep forever — memory is state, not a log); the Postgres deletes are BOUNDED to 5000 rows/tick via PK `(tenant_id, id) IN (SELECT ... LIMIT ?)` subqueries and the R2 sweep is bounded by a 5000-object delete cap + 20-page `list` scan cap, so a backlog drains over successive ticks, emitting a `retention_sweep` audit summary + `orchestrator_retention_deleted{table}` counter (`table ∈ audit_events | plans | artifacts | memory_vectors`; ConversationDO idle-TTL is handled by the DO alarm) — then `runAnomalyScan` (`jobs/anomaly-detector.ts`) which fires `anomaly_detected` events when per-tool error rates exceed a 24h EWMA baseline and `auto_rollback` (sets `canary_weight = 0`) when the offending **canary variant** is the one spiking (`tool_call` rows carry `manifest_variant`, stamped by the react loop from the resolved variant; a bad stable version no longer rolls back a healthy canary). The `enabled` / `min_volume` / `min_rate` / `baseline_factor` thresholds come from each manifest's `spec.anomaly` block (resolved per error-bearing manifest per tick; defaults in `DEFAULT_ANOMALY_CONFIG`), so a noisy manifest can dampen or mute the detector for itself — detection windows stay global. `runContinuousEvalTick` (`jobs/continuous-eval.ts`) online-benchmarks every in-flight canary: it samples recent production inputs (captured as `user_input` on `tool_call` audit rows by the react loop) within the last window, replays each through the canary *version*, scores it with the Workers-AI judge against a generic quality rubric, and emits `judge_score` events tagged `payload.source: 'continuous'` under the canary's tenant. Stateless (time-windowed like the anomaly detector, deterministic per-input hash sampling); replays run under a context scoped to the **canary's own tenant** (not the anonymous `default` tenant) so the sampled tenant's real `user_input` / `args` / `output_preview` never leak into `default`'s audit log — the replay context sets `replay: true`, the react loop stamps `replay: true` onto those `tool_call` rows, and `sampleInputs` excludes them so a replay's input is never re-sampled. The per-tick knobs (`sample_rate` / `max_replays_per_tick` / `window_ms`) come from the optional `CONTINUOUS_EVAL` JSON env var via `parseContinuousEvalOpts(env)` — each field validated and clamped, falling back to `DEFAULT_CONTINUOUS_EVAL_OPTS` so a bad override degrades to defaults rather than disabling the job. The cron parser supports full 5-field syntax including step expressions (`*/N`, `0-30/5`).

## Conventions

- **Composite tenant keys.** Every Postgres table is `(tenant_id, id)` primary key (or a natural tenant-first composite). New tables must follow this. Always scope reads with `WHERE tenant_id = ${...}` via getDb tagged templates; jsonb params are raw objects, never pre-stringified.
- **Feature plugins.** Vertical features that aren't harness core (commerce) ship as a workspace package exporting a `FelixPlugin` (`packages/harness/src/plugins/types.ts`): routes, tool factories, cron tasks, `selfAuthenticatingMounts`, `rateLimitKey`, `bodyLimitBytes`, plus env vars via module augmentation (`packages/commerce/src/env.ts` augments `@felix/harness/env`). Register in `apps/api/src/composition.ts:installedPlugins()`. Boundary rules (enforced by `apps/api/tests/unit/plugin_boundary.test.ts`, three invariants): the harness NEVER imports `@felix/commerce` — only `apps/api` does, and only from the wiring root; package code never escapes its own dir with relative imports — harness seams are consumed as `@felix/harness/<path-from-src>`. Both packages export TS source via `"./*": "./src/*.ts"`; `@felix/api` depends on both, the harness depends on neither. Plugin-owned Postgres migrations stay in the `apps/api/migrations/` dir, named `NNNN_commerce_*` (or the owning plugin's prefix).
- **Zod schemas are `.strict()`** (extra keys rejected). Adding a manifest field requires updating the Zod schema in `packages/harness/src/manifests/schema.ts`.
- **Manifest resolution.** Request-path code (chat, `/v1/chat/completions`, A2A `tasks/send`) MUST call `resolveManifest(env, tenantId, name)` from `packages/harness/src/manifests/resolver.ts`. It walks the 4-layer chain: tenant Postgres → tenant R2 → global R2 → bundled (audit `source` strings keep the legacy `tenant_d1` name). The sync `loadManifest(name)` from `packages/harness/src/manifests/loader.ts` is reserved for system-only call sites with no tenant context (cron, A2A discovery card, MCP default, federation refresh). The resolver walks the full tenant chain for **every** tenant, including `default` — anonymous traffic can still hit `default`-tenant Postgres/R2 overrides.
- **No module-level mutable state** that affects request handling. The `compose(env)` result is cached at the module top in `index.ts` (per isolate) — that's fine because the provider itself is request-agnostic. Anything request-scoped goes through `RequestContext`.
- **Session and memory enums.** `do` / `sqlite` / `agentcore` (alias) / `none` for `spec.memory.checkpointer` (gates which `SessionStore` is built); `vectorize` / `memory` / `agentcore` (alias) / `none` for `spec.memory.store` (long-term semantic memory). `spec.session.strategy`: `full_replay` (default), `windowed:N`, `summarizing:N`, `semantic:N`. Aliases resolve in `packages/harness/src/session/do-session.ts`.
- **Tool transport seam.** Every `Tool` carries an `executor: ToolExecutor` with a `transport` label (`local` / `mcp` / `a2a` / `container` / `queue` / `sandbox` / `browser`). Worker-local tools use `defineTool(...)`, which wraps a handler in a `localExecutor`; non-local tools use `defineToolWithExecutor(...)` and supply their own executor (`McpExecutor`, `A2AExecutor`, `ContainerExecutor`, `QueueExecutor`, `SandboxExecutor`, `BrowserExecutor`, …). The model loop calls `tool.executor.execute(args, ctx)` uniformly and passes `toolCallId` through `ctx` so async transports can pair an eventual `tool_result` to the dispatching turn. Governance wrappers replace `tool.executor` via `wrapExecutor(inner.executor, ...)` so the inner transport survives.
- **Tool errors.** Transport executors throw `ToolError(code, message)` from `packages/harness/src/tools/errors.ts`. `ToolErrorCode` is the closed taxonomy operators alert on (`orchestrator_tool_calls{status='error'}` carries `error_code`); use the existing codes rather than inventing new ones.
- **Federation bundles.** `PolicyBundle` in R2 is Ed25519-signed; the active key is `POLICY_BUNDLE_PUBKEY` (base64 32-byte raw pubkey). Dev allows unsigned bundles with a warning; staging/production require a valid signature.
- **Wrapper deny outputs.** Governance wrappers (policy / limits / guardrails / judge / approvals) return `denyOutput(content, source)` from `tools/types.ts` for their deny paths. Outer wrappers (especially post-call ones) MUST check `isWrapperDeny(output)` before doing extra work — guardrails' output filter does this so it doesn't redact a policy/approval deny string.
- **Cancellation.** Tools that perform fetch / long work should pass `ctx.signal` through (`fetch(url, { signal })`). The same applies to the model client (`{ signal: opts?.signal }` in fetch calls). Without this, wall-clock breaches only block the *next* call instead of cancelling in-flight work.
- **Code style.** Biome with single quotes, semicolons, trailing commas, 100-col width, 2-space indent. `noExplicitAny: warn`, `noUnusedImports: error`.

## Testing topology

Vitest uses two projects (root `vitest.config.ts`):
- `unit` — node pool, `packages/*/tests/**/*.test.ts` + `apps/*/tests/**/*.test.ts` (excluding `apps/api/tests/integration/**`). Pure logic, schema, governance wrappers, fake DOs (and the fake-sql client in `packages/harness/tests/helpers/`), the plugin-boundary guard in `apps/api/tests/unit/`.
- `workers` — `@cloudflare/vitest-pool-workers` (miniflare/workerd), `apps/api/tests/integration/**/*.test.ts`, booted through the `apps/api/src/index.ts` Worker entry. Bindings are declared explicitly in the config rather than read from `wrangler.jsonc` — the bundled workerd doesn't support the wrapped `AI` binding. The Hyperdrive binding points at a real local Postgres (`pnpm db:up`; `felix_test` database): a Node globalSetup resets the schema and applies `apps/api/migrations/` via node-pg-migrate before every run, files run serially (Postgres persists across files, unlike the old per-run miniflare D1), and tests isolate by distinct tenant ids. Direct job/store calls in tests wrap in `withPgContext` (tests/integration/setup.ts) so clients get closed.

When adding integration tests that need a new binding, add it to the `miniflare.bindings` block in `vitest.config.ts` too.

## Claude Code project config

**Git workflow: never commit to `main`, NEVER stack PRs, and prefer feature-based PRs.** Every change lands via a feature branch (`<type>/<slug>`) that branches from `main` and targets `main`; merging is the human gate (see the branch-pr-workflow skill and `.claude/rules/git-workflow.md`; a PreToolUse hook denies commits/pushes on main). Do not branch from another branch/open-PR or use `gh pr create --base <non-main>` — dependent work goes in one PR or waits for the parent to merge. **The unit of a PR is a feature/area, not a single edit:** group related changes (all the fixes for one feature or audit area) into ONE PR rather than opening a separate PR per file or per fix — land a body of related work as a small number of feature-scoped PRs, not dozens of tiny ones (still don't batch *unrelated* features, and never stack). Step-by-step procedures live as project skills under `.claude/skills/` (add-tool, add-pattern-or-model, manifest-schema-change, pg-migration, governance-wrapper, review-quality/security/performance/tests, deploy-runbook, smoke-test, observability, manifest-ops, staging-auth, docs-sync, felix-dev-loop, branch-pr-workflow) — prefer them over improvising a procedure. Subagents `felix-reviewer`, `felix-test-debugger`, and `felix-docs-writer` are in `.claude/agents/`; path-scoped conventions load from `.claude/rules/` when matching files are touched. Docs/OpenAPI must track code: routes appear in `/openapi.json` only via `createRoute` + `.openapi()`, and a Stop hook flags working trees that change API/schema surfaces without touching `packages/{harness,commerce}/docs/` or CLAUDE.md (see the docs-sync skill). Prose docs ship as a separate static Starlight site (`apps/docs` at docs.felix.run, aggregating `packages/*/docs`); the runtime Worker serves only the Scalar reference at `/docs` and 301s the legacy prose routes. Note: `.claude/skills/` (Claude Code config) is unrelated to the `packages/harness/skills/` directory (Felix runtime skills bundled into the Worker).
