# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Felix — a **managed agents harness** on Cloudflare Workers, in the shape Anthropic describes in [Managed Agents](https://www.anthropic.com/engineering/managed-agents). The harness virtualizes three abstractions so each can be swapped without disturbing the others:

- **Session** — an append-only event log per thread. The harness asks a `SessionStrategy` (`full_replay` / `windowed:N` / `summarizing:N` / `semantic:N`) to render the working-set message array for each model call instead of mutating an in-memory history. Anchor messages (`metadata.pinned: true`) are always included across strategies.
- **Pattern / model registries** — `react`, `deep`, `router`, `parallel`, `groupchat`, `reflect`, `plan_execute` and the Anthropic / OpenAI / Workers AI providers self-register at module load; `getPattern(name)` / `getModelProvider(name)` resolve them at build time. Adding a new loop or provider is one `register*(...)` call.
- **ToolExecutor seam** — every `Tool` carries a `transport`-labelled executor (`local` / `mcp` / `a2a` / `container` / `queue` / `sandbox` / `browser`). The model loop dispatches by name; the harness routes to whichever transport the tool was built with. `queue` is the async transport: the executor enqueues a job carrying `tool_call_id`, returns a chatty stub, and a separate consumer writes the eventual `tool_result` back to the session — `wake()` + `tasks/resubscribe` make the resume seamless. `sandbox` (Cloudflare Sandbox SDK) and `browser` (Browser Rendering via `@cloudflare/puppeteer`) front separate Worker adapters as Service bindings; reference adapters live in `examples/sandbox-worker/` and `examples/browser-worker/`.

A `apiVersion: orchestrator/v1` manifest declares the agent — pattern, model (with optional `fallbacks` + `confidence_escalation`), system prompt, tools, skills, MCP servers, A2A peers, containers, queues, sandboxes, browser tools, memory backend, session strategy, execution mode (transient/durable), tools_retrieval, artifacts, reflect config, procedural memory, auth, policies, limits, guardrails (incl. `judges[]`), approvals — and `buildAgent(...)` compiles it to a runnable `Agent`. The runtime is TypeScript on Hono, with Durable Objects, D1, R2, KV, Vectorize, Queues, Workflows, Analytics Engine, and AI Gateway. Public surfaces: `/v1/*` (OpenAI-compatible), `/chat`, `/a2a`, `/mcp`, `/jobs`, `/plans`, `/audit` (+ `/audit/metrics`), `/approvals`, `/manifests` (+ `/:name/canary`, `/:name/rollback`), `/eval`, `/.well-known/agent-card.json`, `/.well-known/jwks.json` (self-issued JWKS when `JWKS_PUBLIC` set), plus the commerce mounts: `/commerce` (Stripe webhook, consents, attribution), `/acp` (Agentic Commerce Protocol; `ACP_API_KEY` bearer, skips JWT middleware), `/brands`, `/shop` + `/widget` (public per-brand storefront + embed), `/structured` + root `robots.txt` / `sitemap.xml` / `.well-known/ai-catalog.json`, `/entities`, `/b2b` (+ `/b2b/billing`), `/geo`. Manifests can be bundled at build time, dropped into R2, or managed per-tenant through the `/manifests` REST surface (append-only versions with active-pointer rollback, optional canary pointers; gated by `manifests:read` / `manifests:write` scopes). Every management route requires a per-surface scope via `requireScope` (`audit:read`, `approvals:read`/`approvals:decide`, `plans:read`, `jobs:read`/`jobs:write`, `eval:read`/`eval:write`, `consent:read`, `geo:write`, `brands:write`, `b2b:write`, `entities:write`); the gate falls open only in dev with no verifiers.

**Felix Commerce** — the agentic-commerce layer, a separate workspace package: **`@felix/commerce`** at `packages/commerce/` (commerce + `entities/` + `geo/` under its `src/`). Built on the harness seams: catalog/cart/checkout tools (cart lives in the session log as a pinned `audit` event, checkout is approval-gated Stripe), ACP merchant endpoint, per-brand D2C storefront manifests + widget, schema.org/GEO discoverability, B2B quote-to-cash on a pluggable entity data-source seam (native/federated/synced; http/mcp connectors), personalization + visual search + dynamic pricing, consent + attribution. The package exports one supported symbol — a **`FelixPlugin`** (`packages/commerce/src/plugin.ts`; contract in `packages/core/src/plugins/types.ts`) bundling routers, tool factories, cron tasks, the `/acp` self-auth mount, rate-limit keying, and the body-size floor — and core wires it in through the single `commercePlugin` entry in `packages/core/src/composition.ts:installedPlugins()`. The package consumes core seams via `@felix/orchestrator/<path>` source exports (no build step; wrangler/vitest/tsc all resolve TS source through package `exports`); commerce env vars merge into `Env` via module augmentation in `packages/commerce/src/env.ts`. Core is otherwise commerce-blind (enforced by `packages/core/tests/unit/plugin_boundary.test.ts`). Detail in `packages/core/docs/internals/commerce.md`.

## Commands

```bash
pnpm install           # pnpm workspace: packages/core (@felix/orchestrator) + packages/commerce (@felix/commerce); root delegates scripts via --filter
cp packages/core/wrangler.example.jsonc packages/core/wrangler.jsonc   # wrangler.jsonc is gitignored (holds account/resource ids); the example is the tracked template
pnpm build:manifests   # YAML → packages/core/src/manifests/bundled.ts + packages/core/src/skills/bundled.ts (REQUIRED before dev/deploy when manifests change)
pnpm build:docs        # packages/core/docs/*.md → packages/core/src/docs/bundled.ts (rendered in-Worker docs site; dev/deploy scripts run it)
pnpm migrate:local     # apply D1 migrations to local SQLite
pnpm migrate:staging   # apply against orchestrator-staging (remote)
pnpm migrate:production # apply against orchestrator-prod (remote)
pnpm dev               # wrangler dev (runs build:manifests first)
pnpm deploy            # wrangler deploy --env production (pnpm deploy:staging for staging)
pnpm test              # vitest — two projects: `unit` (node) and `workers` (miniflare/workerd)
pnpm test -- packages/core/tests/unit/manifest_schema.test.ts   # single file
pnpm test -- --project unit -t "rejects unknown kind"  # single test by name
pnpm typecheck         # tsc --noEmit
pnpm lint              # biome check packages   (lint:fix to auto-apply)
```

`pnpm build:manifests` reads from the core package's own `packages/core/manifests/*.yaml` and `packages/core/skills/*/SKILL.md` directories. If those directories don't exist the bundle is empty; tests stub manifests directly through `parseManifest` / `_clearManifestCaches`.

## Architecture

### Wiring entry points
- `packages/core/src/index.ts` — Worker default export: `fetch`, `scheduled` (cron: federation refresh + `runScheduledJobs` + `sweepOrphanQueueDispatches` + `runAnomalyScan` + `runContinuousEvalTick`, then every installed plugin's `cronTasks` — commerce contributes `abandoned_cart_scan` and `geo_monitor_tick` — each isolated in its own try/catch), `queue` (audit batching). Also re-exports every DO class (`ConversationDO`, `A2ATaskDO`, `ApprovalsDO`, `FederationDO`) and `AgentWorkflow` — wrangler.jsonc references these names.
- `packages/core/src/app.ts` — `createApp({ tools, defaultManifest, plugins })` returns a Hono app. Receives the `ToolProvider` by injection so tests swap in a stub. Mounts the core sub-routers plus `/health` and the agent-card endpoint, then calls each plugin's `routes(app, { tools })` (before `/docs`); plugins also contribute the auth middleware's `selfAuthenticatingMounts`, the rate limiter's `keyResolvers`, and the `bodyLimit` ceiling (`max(1 MB core floor, ...plugin bodyLimitBytes)`). Also registers the unhandled-exception boundary (`app.onError`): `HTTPException`s thrown by routes flow through with their own response; anything else is turned into a 500 `{error:{message:'internal error'}}`, an `orchestrator_unhandled_error` counter, a structured `unhandled_exception` error log (with stack), **and** an `unhandled_error` audit event so tenant operators see it in `/audit` alongside everything else.
- `packages/core/src/composition.ts` — the only place that knows about both the tool catalog and the orchestrator's `ToolProvider` abstraction. Adding a tool = `provider.register('name', () => defineTool({...}))` here; no orchestrator changes needed. Skill-activation tools (`list_skills` / `activate_skill` / `deactivate_skill`) live here because they read `RequestContext`. Also the plugin wiring root: `installedPlugins()` returns the `FelixPlugin[]` (currently just `commercePlugin`) — the ONLY core-side line that names a plugin; `compose` registers each plugin's tools through it.
- `packages/core/src/env.ts` — single source of truth for binding types. Mirrors `wrangler.jsonc`. `parseModelRoutes(env)` parses the `MODEL_ROUTES` JSON map.

### Request-scoped state (AsyncLocalStorage, not parameter threading)
`packages/core/src/context.ts` defines `RequestContext` ({env, auth, limitState, threadId, manifestId, execCtx}). `authMiddleware` (`packages/core/src/auth/middleware.ts`) populates it via `runWithContext` before any route runs and tears it down with `disposeLimitState` in `finally`. Tools and governance wrappers read it through `getContext()` / `requireContext()`. Preserve this pattern when adding cross-cutting concerns — tool authors should never see limit/policy state as an explicit parameter.

`LimitState` carries an `AbortController` that fires on wall-clock breach or request teardown. The signal is plumbed through `ToolInvocationCtx.signal` (peer + MCP fetches honor it) and through `ModelChatOptions.signal` (Anthropic + OpenAI gateway calls honor it). It also carries cumulative `tokens: { input, output }` accumulated from every model response — the react loop checks `checkTokenBudget(limits, manifestId)` before each model call to enforce `max_input_tokens` / `max_output_tokens` caps.

Cron handlers run **outside** `authMiddleware`. `index.ts:scheduled` wraps its body in `runWithContext(buildAnonymousContext(env, ctx), …)` + `disposeLimitState` so audit events actually persist (recordEvent falls back to `console.log` when no context is installed). The queue consumer does not install a RequestContext — it only persists `AuditEvent` rows that were already enqueued by request-scoped code, with a per-row fallback if the batch insert fails so one poison row doesn't starve the queue.

`app.onError` also runs outside any `runWithContext` scope: the auth middleware's `runWithContext` has already unwound by the time Hono routes the exception to the error handler, so `getContext()` returns undefined there. The handler instead uses `recordEventDetached(env, opts, execCtx)` (`packages/core/src/audit/store.ts`) — same pipeline as `recordEvent`, but takes `env` and `execCtx` explicitly off the Hono context instead of reaching for AsyncLocalStorage. Use this helper for any other detached callers that need to emit an audit event.

### Manifest → Agent pipeline
`packages/core/src/manifests/builder.ts:buildAgent` is the single compile step:
1. load + Zod-validate (`schema.ts`, `validate.ts`). `validate.ts` queries `isMultiAgentPattern(name)` on the pattern registry, so cross-field rules (sub_agents required, peers forbidden, durable mode requires single-agent + checkpointed memory, sandboxes/browser_tools forbidden on multi-agent) auto-apply.
2. resolve system prompt (soul / inline / base)
3. resolve `SessionStore` (`getSessionStore(env, memory.checkpointer)`) and `SessionStrategy` (`getSessionStrategy(spec.session.strategy)`)
4. compose skills (`packages/core/src/skills/loader.ts` + per-tenant activation overlay from D1)
5. bind external MCP servers (namespaced) — each remote tool gets an `McpExecutor` (`transport: 'mcp'`)
6. build A2A peer tools (`a2a/client.ts:makePeerTool`) — each gets an `A2AExecutor` (`transport: 'a2a'`); `mcp/client.ts` forwards remote MCP `inputSchema` to the LLM via the `rawInputSchema` escape hatch on `Tool`
7. build container tools (`tools/container-executor.ts:makeContainerTool`) — each `spec.containers[]` entry becomes a tool with a `ContainerExecutor` (`transport: 'container'`); the same auth broker MCP / A2A use supplies the `Authorization` header so raw tokens never enter the executor closure
8. build queue tools (`tools/queue-executor.ts:makeQueueTool`) — each `spec.queues[]` entry resolves a Cloudflare Queue producer from `env[binding]` and wraps it in a `QueueExecutor` (`transport: 'queue'`). Missing bindings fail the build.
8b. build sandbox tools (`tools/sandbox-executor.ts:makeSandboxTool`) — each `spec.sandboxes[]` entry wraps a Service binding in a `SandboxExecutor` (`transport: 'sandbox'`). The bound Worker fronts the Cloudflare Sandbox SDK (see `examples/sandbox-worker/`).
8c. build browser tools (`tools/browser-executor.ts:makeBrowserTool`) — each `spec.browser_tools[]` entry wraps a Service binding in a `BrowserExecutor` (`transport: 'browser'`). The bound Worker fronts `@cloudflare/puppeteer` over Browser Rendering (see `examples/browser-worker/`).
9. resolve sub-agents (multi-agent patterns) and tools (everyone else); auto-inject memory tools when `memory.store` resolves to `vectorize`, `recall_procedure` when `spec.procedural_memory.enabled`, and `fetch_artifact` when `spec.artifacts.enabled`. `PLAN_TOOLS` injection for `deep` lives in `patterns/deep.ts`'s registered adapter, not in the builder.
10. governance pipeline: `mergeWithManifest` → `applyPolicies` → `applyLimits` → `applyGuardrails` → `applyJudges` → `applyApprovals`. Each wrapper replaces `tool.executor` via `wrapExecutor(...)` so the inner transport label survives.
11. `getPattern(spec.pattern)(ctx)` — open registry dispatch. The pattern descriptor picks what it needs from `PatternBuildContext` (`tools`, `subAgents`, `session*`, `limits`, `toolsRetrieval`, `artifacts`, …).
12. if `spec.execution.mode === 'durable'`, wrap the result via `wrapDurableAgent(agent, manifest, env)` — the wrap runs `invoke` inside `env.AGENT_WORKFLOW.create({ params })` so a Worker eviction mid-loop replays the step. Falls back to transient when the binding is absent.

Builder takes `BuildDeps` (env, tools, optional auth/soulLoader/extraTools/subAgentBuilder) — don't reach for module-level singletons. Adding a new pattern or provider is a `registerPattern(...)` / `registerModelProvider(...)` call in the relevant module's bottom matter; `composition.ts` can register additional ones at deploy time without core changes.

### Patterns
No LangGraph. `patterns/react.ts` is a hand-written tool loop bounded by `recursion_limit` (model turns) + the limits wrapper (per-call caps: tool calls, peer hops, wall clock, tokens). Tool dispatch is sequential for deterministic audit ordering. Tools that throw a `ToolError` (`packages/core/src/tools/errors.ts`) are stringified as `[tool error/<code>] <msg>` (code ∈ `invalid_arguments` | `transport_unavailable` | `provider_error` | `timeout` | `user_aborted` | `rate_limited` | `permission_denied` | `internal`) and fed back to the model unless the tool declares `fatal: true` — those terminate the loop with the error message as `final`. Multi-agent patterns each get a dedicated DO for state. Router and parallel call `checkTokenBudget(limits, manifestId)` before each child dispatch; react calls `checkPreflightTokenBudget(...)` (Anthropic's free `/v1/messages/count_tokens`) and falls back to `checkTokenBudget` for routes that don't implement `countTokens`. When `spec.tools_retrieval.enabled`, react filters tools by BGE cosine similarity before each model call. When `spec.artifacts.enabled`, tool results over `threshold_chars` are spilled to R2 (`fetch_artifact` reads them back).

`reflect` (`patterns/reflect.ts`) wraps a react base with a verifier model; below `spec.reflect.threshold` it appends the critique as a synthetic user turn and replays react up to `spec.reflect.max_iterations`. Emits `judge_score` audit events with `payload.source: 'reflect'`. `plan_execute` (`patterns/plan-execute.ts`) splits the loop into a planner model (emits a JSON subtask list), an executor model (runs each subtask in a bounded react sub-loop), and a synthesizer pass (produces the final assistant turn). Emits `plan_step` audit rows with `payload.source: 'plan_execute'`; re-plans up to `spec.plan_execute.max_replans` times when a subtask fails and `replan_on_failure: true`.

`patterns/model.ts` is the AI Gateway model client — Anthropic + OpenAI through `https://gateway.ai.cloudflare.com/v1/${AI_GATEWAY_ACCOUNT_ID}/${AI_GATEWAY_SLUG}/<provider>/...`, Workers AI through the native `env.AI` binding. Every gateway call attaches `cf-aig-authorization: Bearer ${CF_AIG_TOKEN}` when the secret is set (required for slugs with Authenticated Gateway enabled). `streamChat` returns `AsyncGenerator<string, ModelChatResult>` — text deltas yield, the final assistant turn (with any `tool_calls`, `thinking` / `redacted_thinking` blocks, and `usage`) is the generator's return value. The streaming react loop captures it with `await stream.next()` once `done === true`, eliminating the second non-stream `chat()` call earlier versions made. Workers AI SSE reads are buffered across chunk boundaries so split frames don't drop deltas.

`spec.model.fallbacks: string[]` wraps the primary client with `withModelFallbacks` — on `provider_error`/`rate_limited`/`timeout` the chain replays the same call against each fallback id, emits a `model_switch { reason: 'fallback' }` audit, increments `orchestrator_model_switches`. `spec.model.confidence_escalation: { high_id, threshold }` routes low-confidence assistant turns to a higher-capacity model (`reason: 'escalation'`).

Anthropic-specific knobs on `spec.model`: `cache: true` adds `cache_control: { type: 'ephemeral' }` markers to the system prompt, the last tool definition, and the last conversation message (skipped on `thinking` / `redacted_thinking` blocks — Anthropic rejects cache markers there). `thinking_budget` (1024–64000) enables extended thinking, forces `temperature: 1`, and bumps `max_tokens` to at least `budget + 1024`; returned `thinking` / `redacted_thinking` blocks are captured on the assistant message and round-tripped on the next request so tool-result follow-ups don't get rejected. Both are no-ops for OpenAI / Workers AI routes (OpenAI prompt caching is automatic — its `cached_tokens` are subtracted from `prompt_tokens` before populating `TokenUsage.input` so they don't double-count against `max_input_tokens`).

### Session persistence
The `Session` abstraction (`packages/core/src/session/types.ts`) is the harness's external context object. Each `SessionEvent` carries a monotonic `seq`, a `kind` discriminator (`message` / `tool_result` / `tool_call` / `thinking` / `audit`), the message-shaped payload, and optional `metadata`. `ConversationDO` (`packages/core/src/memory/conversation-do.ts`) is keyed by thread id and exposes `GET /events?from&to&limit&kinds`, `GET /head`, `POST /events`, `DELETE /events` — there is no all-or-nothing `/history` dump.

`SessionStore.open(threadId)` returns a per-thread `Session`. The react/deep/groupchat patterns call `await strategy.render(session, incoming, { systemPrompt, model })` to produce the working-set message array, then append new events incrementally via `persistFireAndForget(session, events, { manifestId })` (in `packages/core/src/session/do-session.ts`), which routes through `execCtx.waitUntil` so DO writes don't block LLM steps. Bounded retry (3×, 50ms / 150ms / 450ms) covers transient failures; terminal failures emit a `checkpoint_failure` audit event + `orchestrator_checkpoint_failures` counter rather than being swallowed by `console.warn`.

`SessionStrategy` (`packages/core/src/session/strategies.ts`) decides what the model sees:
- `full_replay` (default) — every prior message, exactly as in the legacy checkpointer.
- `windowed:N` — keep the last N events; drop the rest.
- `summarizing:N` — keep the last N raw events, model-summarize everything older into a synthetic system message. The summary is cached as a `kind: 'audit'` event with `metadata: { type: 'session_summary', covers_to_seq: N }`, so steady-state rendering skips the model call until new events cross the keep boundary. Degrades to windowed when no model is supplied or the summarizer call throws — never fails the request.
- `semantic:N` (`packages/core/src/session/semantic-strategy.ts`) — embed the current user message via BGE and pull the top-N most relevant prior events from the session log instead of the most recent N. Anchor messages (`metadata.pinned === true`) are always included regardless of score.

`Session.wake()` (`packages/core/src/session/types.ts:analyzeWake`) is the resume primitive — Anthropic's Managed Agents `wake(sessionId)`. Read-only analysis of the event log that returns `{ fresh, headSeq, pendingToolCalls, endedOnAssistant }`. Used by the A2A `tasks/resubscribe` method (`packages/core/src/a2a/server.ts:tasksResubscribe`) to let a reconnecting client pick up: terminal tasks replay events + emit cached output; in-progress tasks replay events + emit a `continue_hint` directing the client to issue `tasks/sendSubscribe` with the same task id so the next loop step picks up from the resume point.

Routing:
- `POST /chat`, `POST /chat/stream` — caller supplies `thread_id` in the body.
- `POST /v1/chat/completions` — opt-in via `x-thread-id` header; absent, each request is stateless (`tenant:openai-<uuid>`).
- A2A `tasks/send` / `sendSubscribe` — the task id becomes the thread id (`tenant:a2a-<taskId>`).
- **router** forwards `threadId` to its chosen child. **parallel** strips it (prevents race writes). **groupchat** owns the transcript itself: hydrates from + appends to the parent threadId, deliberately does **not** forward it to children (they'd race-write the same DO).

### Persistence layout
- D1 (`DB`, migrations `0001`–`0018`) — harness core (`0001`–`0005`): audit, plans, jobs, approvals, skill_activation, oauth_token_cache, manifests + manifest_active, eval_datasets + eval_dataset_items + eval_runs. Commerce (`0006`–`0018`): products/orders/order_items, acp_checkout_sessions, brands + brand_domains, data_sources, B2B accounts/buyers/quotes/quote_items/invoices/contract_prices/billing_settings, geo_queries/geo_observations, consents/order_attribution, customers/customer_sessions/behavior_events/abandoned_carts, pricing_rules/competitor_prices. **Tenant-first composite primary keys** (`(tenant_id, id)` in the common case; natural composites like `(tenant_id, name, version)` for manifests, `(tenant_id, thread_id)` for carts; `brand_domains` is host-keyed by design) with `(tenant_id, ts DESC)` indexes — always include the tenant in WHERE clauses. `0002_harden.sql` rewrote `jobs` to PK `(tenant_id, name)` (closes a cross-tenant leak) and adds a partial `idx_jobs_next_run` index. `0003_manifests.sql` adds `manifests` (append-only version log) and `manifest_active` (pointer) so tenants can manage their own manifests. `0004_eval.sql` adds the eval harness tables. `0005_manifest_canary.sql` extends `manifest_active` with `canary_version` + `canary_weight` columns; the resolver picks stable vs canary by SHA-256 hash of `(tenant_id, thread_id, manifest_name, stable_version, canary_version)`. `oauth_token_cache.access_token` is AES-256-GCM encrypted at rest via `OAUTH_CACHE_KEY` (`packages/core/src/security/at-rest.ts`); dev falls back to plaintext with a warning, staging/production require the secret.
- KV (`CACHE`) — JWKS cache, outbound OAuth token cache, manifest cache.
- R2 (`BUNDLES`) — federation `PolicyBundle`, per-tenant manifest overrides at `manifests/<tenant_id>/<name>.json`, global manifest overrides at `manifests/<name>.json`, and (when `spec.artifacts.enabled`) artifact spills at `artifacts/<tenant_id>/<thread_id>/<tool_call_id>.json`. Tenant D1 wins over tenant R2 wins over global R2 wins over bundled.
- Vectorize (`MEMORY_VEC`) — semantic memory, JIT tool retrieval embeddings, `semantic:N` session strategy, and procedural memory (all share the same 768-dim BGE pipeline, filtered by `{ tenant, manifest, kind }`). The builder auto-injects `memory_remember` / `memory_recall` when a manifest declares `memory.store: vectorize` (or legacy `agentcore` alias), and `recall_procedure` when `spec.procedural_memory.enabled`.
- Queue (`AUDIT_QUEUE`) — audit events fan out via the producer in `audit/store.ts`; the batched consumer in `index.ts:queue` flushes ≤50 events per pull via `DB.batch()`.
- Analytics Engine (`METRICS`) — `recordCounter`/`recordHistogram` write to `env.METRICS.writeDataPoint(...)` when the binding is present; stdout fallback continues for dev parity.
- Workflows (`AGENT_WORKFLOW`) — when a manifest declares `execution.mode: durable`, the builder wraps the agent in a `DurableAgent` that runs `invoke` inside `env.AGENT_WORKFLOW.create({ params })`. The workflow class is `AgentWorkflow` in `packages/core/src/workflows/agent-workflow.ts`.

### Durable Objects
- `ConversationDO` — one per thread id.
- `A2ATaskDO` — one per `tenant#task` id, owns A2A task lifecycle state.
- `ApprovalsDO` — serializes concurrent `decide` writes on the same approval id; `/approvals/:id/decide` routes through it.
- `FederationDO` — cron-refreshed singleton holding the active `PolicyBundle` from R2.

### Auth
Inbound JWT verifiers (`auth/jwt.ts`) are configured entirely through the single `JWT_VERIFIERS` env var — there are no provider-specific vars. It's a comma-separated list; each verifier is whitespace-separated `<scheme> <issuer> [audience]` where `scheme` is `access` (Cloudflare Access JWKS at `https://<issuer>/cdn-cgi/access/certs`) or `cognito` (standard OIDC JWKS at `<issuer>/.well-known/jwks.json`). `parseVerifiers(env)` skips malformed/unknown-scheme entries; an empty value means no verifiers (fail-closed in prod, anonymous in dev). Anonymous context is always populated — the route/manifest decides whether `allow_anonymous` is OK. Outbound OAuth client-credentials tokens are cached in D1 (encrypted at rest, see persistence layout).

### Cron + jobs
`triggers.crons: ["*/10 * * * *"]` → `scheduled()` runs federation refresh, then `runScheduledJobs` (`jobs/cron.ts`), then `runAnomalyScan` (`jobs/anomaly-detector.ts`) which fires `anomaly_detected` events when per-tool error rates exceed a 24h EWMA baseline and `auto_rollback` (sets `canary_weight = 0`) when the offending **canary variant** is the one spiking (`tool_call` rows carry `manifest_variant`, stamped by the react loop from the resolved variant; a bad stable version no longer rolls back a healthy canary). The `enabled` / `min_volume` / `min_rate` / `baseline_factor` thresholds come from each manifest's `spec.anomaly` block (resolved per error-bearing manifest per tick; defaults in `DEFAULT_ANOMALY_CONFIG`), so a noisy manifest can dampen or mute the detector for itself — detection windows stay global. `runContinuousEvalTick` (`jobs/continuous-eval.ts`) online-benchmarks every in-flight canary: it samples recent production inputs (captured as `user_input` on `tool_call` audit rows by the react loop) within the last window, replays each through the canary *version*, scores it with the Workers-AI judge against a generic quality rubric, and emits `judge_score` events tagged `payload.source: 'continuous'` under the canary's tenant. Stateless (time-windowed like the anomaly detector, deterministic per-input hash sampling); replays run under the anonymous cron context so their own `tool_call` rows land under `default` and are never re-sampled (`listActiveCanaries` skips `default`). The per-tick knobs (`sample_rate` / `max_replays_per_tick` / `window_ms`) come from the optional `CONTINUOUS_EVAL` JSON env var via `parseContinuousEvalOpts(env)` — each field validated and clamped, falling back to `DEFAULT_CONTINUOUS_EVAL_OPTS` so a bad override degrades to defaults rather than disabling the job. The cron parser supports full 5-field syntax including step expressions (`*/N`, `0-30/5`).

## Conventions

- **Composite tenant keys.** Every D1 table is `(tenant_id, id)` primary key. New tables must follow this. Always scope reads with `WHERE tenant_id = ?`.
- **Feature plugins.** Vertical features that aren't harness core (commerce) ship as a workspace package exporting a `FelixPlugin` (`packages/core/src/plugins/types.ts`): routes, tool factories, cron tasks, `selfAuthenticatingMounts`, `rateLimitKey`, `bodyLimitBytes`, plus env vars via module augmentation (`packages/commerce/src/env.ts` augments `@felix/orchestrator/env`). Register in `composition.ts:installedPlugins()`. Boundary rules (enforced by `packages/core/tests/unit/plugin_boundary.test.ts`): core imports `@felix/commerce` only from the wiring root; package code never escapes its own dir with relative imports — core seams are consumed as `@felix/orchestrator/<path-from-src>`. Both packages export TS source via `"./*": "./src/*.ts"`; the core↔commerce workspace dependency cycle is deliberate and fine (single compile, pnpm links both). Plugin-owned D1 migrations stay in the core `packages/core/migrations/` dir, named `NNNN_commerce_*` (or the owning plugin's prefix).
- **Zod schemas are `.strict()`** (extra keys rejected). Adding a manifest field requires updating the Zod schema in `packages/core/src/manifests/schema.ts`.
- **Manifest resolution.** Request-path code (chat, `/v1/chat/completions`, A2A `tasks/send`) MUST call `resolveManifest(env, tenantId, name)` from `packages/core/src/manifests/resolver.ts`. It walks the 4-layer chain: tenant D1 → tenant R2 → global R2 → bundled. The sync `loadManifest(name)` from `packages/core/src/manifests/loader.ts` is reserved for system-only call sites with no tenant context (cron, A2A discovery card, MCP default, federation refresh). The resolver walks the full tenant chain for **every** tenant, including `default` — anonymous traffic can still hit `default`-tenant D1/R2 overrides.
- **No module-level mutable state** that affects request handling. The `compose(env)` result is cached at the module top in `index.ts` (per isolate) — that's fine because the provider itself is request-agnostic. Anything request-scoped goes through `RequestContext`.
- **Session and memory enums.** `do` / `sqlite` / `agentcore` (alias) / `none` for `spec.memory.checkpointer` (gates which `SessionStore` is built); `vectorize` / `memory` / `agentcore` (alias) / `none` for `spec.memory.store` (long-term semantic memory). `spec.session.strategy`: `full_replay` (default), `windowed:N`, `summarizing:N`, `semantic:N`. Aliases resolve in `packages/core/src/session/do-session.ts`.
- **Tool transport seam.** Every `Tool` carries an `executor: ToolExecutor` with a `transport` label (`local` / `mcp` / `a2a` / `container` / `queue` / `sandbox` / `browser`). Worker-local tools use `defineTool(...)`, which wraps a handler in a `localExecutor`; non-local tools use `defineToolWithExecutor(...)` and supply their own executor (`McpExecutor`, `A2AExecutor`, `ContainerExecutor`, `QueueExecutor`, `SandboxExecutor`, `BrowserExecutor`, …). The model loop calls `tool.executor.execute(args, ctx)` uniformly and passes `toolCallId` through `ctx` so async transports can pair an eventual `tool_result` to the dispatching turn. Governance wrappers replace `tool.executor` via `wrapExecutor(inner.executor, ...)` so the inner transport survives.
- **Tool errors.** Transport executors throw `ToolError(code, message)` from `packages/core/src/tools/errors.ts`. `ToolErrorCode` is the closed taxonomy operators alert on (`orchestrator_tool_calls{status='error'}` carries `error_code`); use the existing codes rather than inventing new ones.
- **Federation bundles.** `PolicyBundle` in R2 is Ed25519-signed; the active key is `POLICY_BUNDLE_PUBKEY` (base64 32-byte raw pubkey). Dev allows unsigned bundles with a warning; staging/production require a valid signature.
- **Wrapper deny outputs.** Governance wrappers (policy / limits / guardrails / judge / approvals) return `denyOutput(content, source)` from `tools/types.ts` for their deny paths. Outer wrappers (especially post-call ones) MUST check `isWrapperDeny(output)` before doing extra work — guardrails' output filter does this so it doesn't redact a policy/approval deny string.
- **Cancellation.** Tools that perform fetch / long work should pass `ctx.signal` through (`fetch(url, { signal })`). The same applies to the model client (`{ signal: opts?.signal }` in fetch calls). Without this, wall-clock breaches only block the *next* call instead of cancelling in-flight work.
- **Code style.** Biome with single quotes, semicolons, trailing commas, 100-col width, 2-space indent. `noExplicitAny: warn`, `noUnusedImports: error`.

## Testing topology

Vitest uses two projects (root `vitest.config.ts`):
- `unit` — node pool, `packages/*/tests/**/*.test.ts` (excluding `packages/core/tests/integration/**`). Pure logic, schema, governance wrappers, fake DOs.
- `workers` — `@cloudflare/vitest-pool-workers` (miniflare/workerd), `packages/core/tests/integration/**/*.test.ts`. Bindings are declared explicitly in the config rather than read from `wrangler.jsonc` — the bundled workerd doesn't support the wrapped `AI` binding, so integration tests stub it via a service worker.

When adding integration tests that need a new binding, add it to the `miniflare.bindings` block in `vitest.config.ts` too.

## Claude Code project config

Step-by-step procedures live as project skills under `.claude/skills/` (add-tool, add-pattern-or-model, manifest-schema-change, d1-migration, governance-wrapper, review-quality/security/performance/tests, deploy-runbook, smoke-test, observability, manifest-ops, staging-auth, docs-sync, felix-dev-loop) — prefer them over improvising a procedure. Subagents `felix-reviewer`, `felix-test-debugger`, and `felix-docs-writer` are in `.claude/agents/`; path-scoped conventions load from `.claude/rules/` when matching files are touched. Docs/OpenAPI must track code: routes appear in `/openapi.json` only via `createRoute` + `.openapi()`, and a Stop hook flags working trees that change API/schema surfaces without touching `packages/core/docs/` or CLAUDE.md (see the docs-sync skill). Note: `.claude/skills/` (Claude Code config) is unrelated to the `packages/core/skills/` directory (Felix runtime skills bundled into the Worker).
